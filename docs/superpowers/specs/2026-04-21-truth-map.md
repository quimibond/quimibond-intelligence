# Truth Map — Quimibond Multi-Source Data Canonicity

**Fecha:** 2026-04-20
**Propósito:** declarar, para cada entidad y campo crítico, QUÉ fuente es la verdad canónica. Base del motor de reconciliación (Fase 3+: Supabase deja de ser mirror-of-Odoo y pasa a ser capa de reconciliación + inteligencia).
**Estado:** borrador para revisión del usuario. Donde no hay regla clara marcamos `⚠ TBD — user decision`.
**Evidencia:** conteos citan `audit_tolerances`, `reconciliation_issues`, `invoice_bridge`, `companies`, `entities` vivos al 2026-04-20 en `tozqezmivpblmcubmnpi` y spec 07 (`2026-04-21-supabase-audit-07-desaprovechado.md`).

---

## Fuentes en juego

- **Odoo ERP (qb19 addon)** — fuente operativa: órdenes venta/compra, inventario, entregas (stock.picking), manufactura (mrp.production), pagos registrados (account.payment), facturas contables (account.move), plan de cuentas, balances bancarios, tipos de cambio, empleados, usuarios. Frescura: push cada 1h (sample 2026-04-21 01:19 UTC). Identificador natural: `odoo_*_id`.
- **SAT vía Syntage** — fuente fiscal legal: CFDIs Ingreso/Egreso/Nómina (tipos I/E/N), Complementos de Pago (tipo P), retenciones ISR/IVA emitidas a Quimibond, declaraciones mensuales, contabilidad electrónica (catálogo/balanza), opinión de cumplimiento propia, flags 69B para terceros (`emisor_blacklist_status` / `receptor_blacklist_status`). Frescura: webhook + pull-sync (último sync 2026-04-20 08:08 UTC). Identificador natural: `uuid` (Folio Fiscal).
- **Gmail** — fuente conversacional: emails, hilos, adjuntos (XML/PDF de CFDI/POs/facturas). Identificador: `gmail_message_id`. Poblada por `/api/pipeline/sync-emails`.
- **Claude AI (Knowledge Graph)** — fuente sintética: `entities` (person/company/product/machine/raw_material), `facts` (information/commitment/request/complaint/price), `entity_relationships`. Extraída por `/api/pipeline/analyze`. No es autoritativa por sí sola — es señal.
- **Manual overrides** — fuente humana explícita: `invoice_bridge_manual`, `payment_bridge_manual`, `products_fiscal_map` (top 20 SKUs confirmados a UNSPSC), futuras reconciliaciones humanas via `reconcile_invoice_manually()` / `reconcile_payment_manually()`.
- **Bancos (futuro, out of scope)** — statement bancario directo no integrado hoy. Placeholder para reconciliación de pagos reales vs Odoo/SAT.

---

## Principios

1. **Fiscal > Operativo** cuando el campo tiene implicación fiscal/legal (UUID, estado SAT vigente/cancelado, total_mxn declarado, fecha_timbrado, RFC receptor, tipoCambio del CFDI). Syntage gana porque es lo que el SAT reconoce como verdad.
2. **Operativo > Fiscal** cuando el campo es de gestión interna (salesperson, commitment_date, margen, inventario, costo estándar, payment_term operativo, ruta de entrega, centro de producción). Odoo gana porque el SAT no tiene esa vista.
3. **Manual > Automático** siempre. Una entrada en `invoice_bridge_manual` / `payment_bridge_manual` / `products_fiscal_map` **supera** cualquier matching automático, incluso UUID exact. Humano cierra el loop.
4. **Más reciente > más viejo** con tie-break por timestamp (`synced_at` / `write_date` / `updated_at`). En caso de empate, se prefiere la fuente con mayor autoridad por el principio 1/2.
5. **Conflicto sin regla → escalar**. Si dos fuentes autoritativas difieren y no hay regla explícita, la fila entra a `reconciliation_issues` o a un dashboard de "inbox del usuario" antes de decidir.
6. **UUID es natural key invariable**. Una vez un CFDI tiene UUID, ese UUID manda. No se deduplica por otro campo.
7. **Tolerancias declaradas en `audit_tolerances` son ley**. Si una diff está dentro de tolerancia, no cuenta como conflicto (ver Matriz).

## Tolerancias vigentes (audit_tolerances — 2026-04-20)

Estas son las invariantes ya configuradas como "truth contracts" ejecutándose en el harness de auditoría. Son parte del Truth Map.

| invariant_key | abs_tolerance | pct_tolerance | Significado operativo |
|---|---|---|---|
| `account_balances.cogs_accounts_balance` | 1.00 MXN | 0.05% | Suma balance cuentas COGS mensuales reconcilia con detalle contable |
| `account_balances.inventory_accounts_balance` | 1.00 MXN | 0.05% | Cuentas de inventario reconcilian |
| `account_balances.revenue_accounts_balance` | 1.00 MXN | 0.05% | Cuentas de ingreso reconcilian |
| `bank_balances.native_balance_per_journal` | 0.05 MXN | 0.01% | Saldo banco diario matchea a centavo |
| `invoice_lines.sum_subtotal_signed_mxn` | 0.50 MXN | 0.50% | Suma líneas factura = total factura (margen de FX) |
| `order_lines.sum_subtotal_mxn` | 0.50 MXN | 0.50% | Suma líneas pedido = total pedido (margen de FX) |

**Nota:** todas las tolerancias MXN usan el tipo de cambio del documento, no del momento del audit. `0.50%` compensa drift de FX intra-día.

**Bridge live:** `invoice_bridge` view (96,512 filas totales — recalculado 2026-04-20)
- `match_uuid` (UUID exact): **8,258**
- `match_composite` (RFC+fecha+total): **7,412**
- `odoo_only` (Odoo sin SAT): **12,030**
- `syntage_only` (SAT sin Odoo): **68,671**
- `ambiguous`: **141**

**Discrepancias hoy (spec 07 §4 / reconciliation_issues):**
- `sat_only_cfdi_issued`: **30,770 total / 9,985 open** — CFDIs emitidos en SAT sin Odoo (subset post-2021 es real problema; pre-2021 son histórico pre-Odoo).
- `sat_only_cfdi_received`: **20,594 total / 10,918 open** — proveedor CFDI sin factura Odoo.
- `cancelled_but_posted`: **97 open** — SAT canceló, Odoo sigue posted.
- `amount_mismatch`: **24 total / 15 open** — fuera de tolerancia `0.50 MXN / 0.50%`.
- `posted_but_sat_uncertified`: **177 total / 134 open** — Odoo posted, Syntage aún no tiene timbre.
- Odoo posted sin `cfdi_uuid`: **13,775 rows** (post-cleanup Fase 2 del bug `_build_cfdi_map`).

**Proposed reconciliation rule:**
- Si `invoice_bridge_manual` link existe → usa ese.
- Si `uuid_exact` → usar Syntage para campos fiscales, Odoo para campos operativos (tabla arriba).
- Si `match_composite` y amount diff ≤ tolerancia → `match_quality='medium'`, tratar como uuid_exact para fiscal.
- Si `odoo_only AND invoice_date >= 2021-01-01 AND state='posted'` → `missing_sat_timbrado` (flag, no conflicto).
- Si `odoo_only AND state='cancel'` → no conflicto (factura cancelada antes de timbrar).
- Si `syntage_only AND fecha_timbrado >= 2021-01-01` → `missing_odoo` (entrar a inbox operador).
- Si `syntage_only AND fecha_timbrado < 2021-01-01` → `resolution='historical_pre_odoo'` (ya aplicado, 30,404 rows).
- Si `amount diff > tolerancia AND within FX tolerance band` → flag `amount_diff_post_fx` en vez de `amount_mismatch`.

## Reglas por fuente de conflicto (resumen ejecutivo)

Cuando dos fuentes chocan, aplica en este orden:

1. **Override manual** (`invoice_bridge_manual`, `payment_bridge_manual`, `products_fiscal_map`, `facts.verified=true`) → wins siempre.
2. **Principio de dominio:**
   - Campo fiscal (UUID, estado SAT, total MXN fiscal, tipo_cambio CFDI, método/forma de pago, claveProdServ, RFC, opinion_cumplimiento, 69B) → **SAT**.
   - Campo operativo (salesperson, commitment_date, state Odoo, margin, stock, reorder, payment_term, list_price, standard_price, journal, conciliación bancaria, carrera de órdenes) → **Odoo**.
   - Campo relacional / comunicación (sentiment, relationship_score, commitment facts, domain) → **Claude** (con confidence).
3. **Más reciente** cuando el dominio es ambiguo (tie-break por `synced_at`/`write_date`/`updated_at`).
4. **Tolerancias `audit_tolerances`** — si la diff cae dentro de la tolerancia, NO es conflicto.
5. **Escalar a inbox** si no hay regla clara — nunca silenciar.
